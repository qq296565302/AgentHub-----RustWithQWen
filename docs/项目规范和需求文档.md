## 1. 项目概述

### 1.1 产品定位

> **AgentHub** 是一个安全、可编程、Skill 驱动的本地 AI 执行引擎，专为中小研发团队设计。它让团队无需招聘 AI 工程师，即可在本地部署企业级的 AI 编程协作者。

### 1.2 核心价值

| 价值维度 | 说明 |
|---------|------|
| **安全合规** | 代码 100% 留在本地，符合企业数据安全要求 |
| **开箱即用** | 预置研发场景专用 Skill（代码解释、测试生成等） |
| **高度可定制** | 支持自定义 Skill，适配团队特定需求 |
| **多前端支持** | 通过统一 API 支持 VS Code、Web、CLI 等多种交互方式 |
| **轻量部署** | 单二进制文件（≤50MB），可运行在开发机或 2C4G 云服务器 |

### 1.3 目标用户

| 用户类型 | 使用场景 | 核心诉求 |
|---------|---------|---------|
| **中小软件公司** | 提升研发效率，降低代码质量风险 | 安全、易用、提效 |
| **独立开发者** | 个人项目辅助开发 | 轻量、免费、强大 |
| **技术团队负责人** | 统一团队开发规范 | 可定制、可审计 |

### 1.4 与竞品对比

| 特性 | AgentHub | GitHub Copilot | Continue.dev | Sourcegraph Cody |
|------|----------|----------------|--------------|------------------|
| **本地部署** | ✅ | ❌ | ✅ | ❌（企业版可私有化） |
| **代码不出本地** | ✅ | ❌ | ✅ | ❌ |
| **可定制 Skill** | ✅ | ❌ | ✅ | ❌ |
| **多前端支持** | ✅ | ❌ | ❌ | ❌ |
| **安全审计** | ✅ | ❌ | ❌ | ✅（企业版） |
| **开源核心** | ✅（计划） | ❌ | ✅ | ✅（部分） |

#### 2.4.3 技术决策说明

**为什么选择 tokio 而非 async-std？**
```toml
# Cargo.toml
[dependencies]
tokio = { version = "1", features = ["full"] }  # 推荐
axum = "0.7"  # 基于 tokio 生态
```
- ✅ axum Web 框架原生基于 tokio
- ✅ 社区更活跃，文档更完善
- ✅ 与 Ollama 客户端兼容性更好
- ✅ 异步运行时成熟度更高

**为什么使用 anyhow + thiserror 组合？**
```rust
// src/error.rs - 统一错误处理
use thiserror::Error;

#[derive(Error, Debug)]
pub enum AgentHubError {
    #[error("Skill error: {0}")]
    Skill(#[from] SkillError),
    
    #[error("Security violation: {0}")]
    Security(#[from] SecurityError),
    
    #[error("LLM error: {0}")]
    LLM(#[from] LlmError),
    
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
    
    #[error("Configuration error: {0}")]
    Config(String),
}

pub type Result<T> = std::result::Result<T, AgentHubError>;
```

**Phase 1 推荐的错误上下文传播模式（anyhow::Context）**

> **为什么需要 Context？**
> - 扁平错误难以定位深层问题（如"文件读取失败"但不知道是哪个文件）
> - `anyhow::Context` 可为每个错误添加上下文信息
> - 错误链可追溯：`原始错误 → 上下文1 → 上下文2 → ...`

**推荐写法（在业务代码中使用）**：

```rust
// src/skill/builtins/code_explainer.rs
use anyhow::Context;

pub async fn explain_file(&self, file_path: &Path) -> Result<String, AgentHubError> {
    // ✅ 正确做法：使用 .context() 添加上下文
    let code = std::fs::read_to_string(file_path)
        .with_context(|| format!("Failed to read source file: {}", file_path.display()))?;
    
    let ast = self.parser.parse(&code)
        .context("Failed to parse AST from source code")?;
    
    let functions = extract_functions(&ast)
        .context("Failed to extract function definitions")?;
    
    // ... 其他逻辑
    
    Ok(result)
}
```

**错误输出示例（带完整上下文链）**：

```
❌ 错误: Failed to process code explanation

Caused by:
   0: Failed to read source file: /home/user/project/src/main.rs
   1: No such file or directory (os error 2)
   
   堆栈跟踪:
      src/skill/builtins/code_explainer.rs:42 - explain_file()
      src/api/cli/commands.rs:78 - execute_explain()
      src/api/cli/repl.rs:105 - execute_command()
```

**统一错误转换宏（简化重复代码）**：

```rust
// src/error/context.rs
use std::path::Path;
use anyhow::{Context, Result};

/// 为文件操作添加标准上下文的辅助宏
macro_rules! with_file_context {
    ($expr:expr, $path:expr, $operation:expr) => {
        $expr.with_context(|| format!(
            "{} failed for file: {}",
            $operation,
            $path.display()
        ))
    };
}

/// 使用示例
let content = with_file_context!(
    std::fs::read_to_string(path),
    path,
    "Reading source file"
)?;
```

**日志中的错误记录最佳实践**：

```rust
// 在 API 层记录完整错误信息
match result {
    Ok(response) => info!("✅ Skill '{}' executed successfully", skill_name),
    Err(e) => {
        error!(
            "❌ Skill '{}' execution failed:\n{:#?}",
            skill_name,
            e  // {:#?} 格式化显示完整错误链
        );
        
        // 同时记录到审计日志
        audit_log.error_message = Some(format!("{:#}", e));
    }
```

**Guardrail 接口**：

```rust
pub trait Guardrail {
    fn name(&self) -> &'static str;
    fn inspect_input(&self, input: &str) -> Result<(), SecurityError>;
    fn inspect_output(&self, output: &str) -> Result<String, SecurityError>;
}

pub struct SecurityPipeline {
    layers: Vec<Box<dyn Guardrail + Send + Sync>>,
}

impl SecurityPipeline {
    pub fn new() -> Self {
        Self {
            layers: vec![
                Box::new(PromptInjectionGuard::default()),
                Box::new(PIIRedactionGuard::default()),
                Box::new(SensitiveWordFilter::default()),
                Box::new(CodeInjectionGuard::default()),
            ],
        }
    }
    
    pub fn process_input(&self, input: &str) -> Result<(), SecurityError> {
        for layer in &self.layers {
            layer.inspect_input(input)?;
        }
        Ok(())
    }
    
    pub fn process_output(&self, output: &str) -> Result<String, SecurityError> {
        let mut result = output.to_string();
        for layer in &self.layers {
            result = layer.inspect_output(&result)?;
        }
        Ok(result)
    }
}
```

## 5. 安全规范

### 5.1 安全原则（Non-Negotiable）

1. **数据不出本地**：默认禁止所有外网请求
2. **最小权限原则**：Skill 默认只读，写操作需显式授权
3. **安全默认开启**：所有防护策略默认启用且不可全局关闭
4. **审计必存**：所有操作必须记录审计日志
5. **输入验证**：所有外部输入必须经过 SecurityPipeline

### 5.1.1 增强安全机制（建议补充）

>  **完整实现（SecurityPipeline、SecurityCheckResult 分级、PII 检测、渐进式安全基线）见《开发实战指南》第 3 节 Day 2**

**核心设计**：
| 组件 | 职责 |
|------|------|
| `SecurityCheckResult` | 分级警告：PASS / WARN(可继续) / BLOCK(拒绝) |
| `SecurityPipeline` | 组合检查链：InputValidator  PromptInjection  PII  FileAccess |
| `PromptInjectionDetector` | 基线：正则匹配；增强：Unicode 规范化 + 零宽字符过滤 + LLM 语义分析 |
| `PIIDetector` | 正则检测 email/phone/IP/密钥/Token 等敏感信息 |

** 安全局限性声明**：基于正则的防护属于基线层，无法防御 Unicode 同形字、零宽字符注入、语义级攻击。生产环境必须启用增强安全层。

```rust
// src/skill/builtins/code_explainer.rs
use serde_json::json;
use crate::skill::{Skill, SkillResult, ExecutionContext, Result};

pub struct CodeExplainerSkill;

impl Skill for CodeExplainerSkill {
    fn name(&self) -> &str {
        "code.explainer"
    }
    
    fn version(&self) -> &str {
        "1.0.0"
    }
    
    async fn execute(
        &self,
        params: serde_json::Value,
        ctx: &ExecutionContext,
    ) -> Result<SkillResult> {
        // 1. 提取参数
        let file_path = params["file_path"]
            .as_str()
            .ok_or(SkillError::InvalidInput("missing file_path"))?;
        
        // 2. 安全读取文件（在 workspace_dir 内）
        let code = ctx.file_access.read_file(file_path).await?;
        
        // 3. 使用 tree-sitter 解析 AST
        let language = detect_language(file_path);
        let ast = parse_code(&code, language)?;
        
        // 4. 构建优化的 Prompt（结合 AST 信息）
        let prompt = build_explanation_prompt(&ast, &code);
        
        // 5. 调用 LLM（带缓存）
        let explanation = ctx.llm_client
            .complete_with_cache(&prompt)
            .await?;
        
        // 6. 返回结果
        Ok(SkillResult {
            output: json!({
                "explanation": explanation,
                "language": language,
                "complexity_score": calculate_complexity(&ast),
            }),
            files_created: vec![],
            warnings: vec![],  // 如有安全警告会在这里
        })
    }
}

fn build_explanation_prompt(ast: &ASTNode, code: &str) -> String {
    format!(
        "请解释以下代码的逻辑和潜在风险：\n\n\
         语言：{}\n\
         代码结构：{}\n\n\
         代码内容：\n```\n{}\n```",
        ast.language,
        ast.structure_summary(),
        code
    )
}
```

```typescript
// TypeScript SDK（供前端使用）
export class AgentHubClient {
    private baseUrl: string;
    
    constructor(baseUrl: string = "http://localhost:8080") {
        this.baseUrl = baseUrl;
    }
    
    async invoke(skillName: string, input: any, userId: string): Promise<any> {
        const response = await fetch(`${this.baseUrl}/v1/invoke`, {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ skill_name: skillName, input, user_id: userId })
        });
        
        if (!response.ok) {
            throw new Error(`API error: ${response.status}`);
        }
        
        return await response.json();
    }
    
    async listSkills(interfaceType?: string): Promise<SkillInfo[]> {
        const url = interfaceType 
            ? `${this.baseUrl}/v1/skills?interface=${interfaceType}`
            : `${this.baseUrl}/v1/skills`;
        
        const response = await fetch(url);
        return await response.json();
    }
}
```

## 9. 性能优化策略
## 9. 性能优化策略

>  **完整实现（LRU Cache、并发控制、增量分析、基准测试脚本、性能回归 CI）见《开发实战指南》第 9 节**

### 9.1 LLM 调用缓存（关键优化）

**方案**：基于 `input_hash(SHA256) + skill_name` 的 LRU 缓存，TTL 带抖动防止雪崩。

| 参数 | 默认值 | 说明 |
|------|--------|------|
| max_entries | 1000 | 最大缓存条目 |
| ttl_seconds | 3600 | 1 小时过期（含 10% 抖动）|

**NFR-002 达成条件**：启用 Cache 后，首次请求 3s，缓存命中 0.1s。

### 9.2 并发控制策略

- **信号量限流**：最大并发 LLM 调用数可配置（默认 4）
- **请求队列**：超出限制时排队等待，超时返回友好错误

### 9.3 增量分析模式

**创新点**：对已分析文件，仅将变更部分（diff）发送给 LLM，大幅减少 Token 消耗。

### 9.4 性能基线数据

| 操作 | Qwen3-4B (本地) | Qwen3-0.5B (快速) |
|------|-------------------|---------------------|
| 解释 50 行 Rust | ~2.3s | ~0.8s |
| 解释 100 行 Rust | ~3.5s | ~1.2s |
| 测试生成 (50 行) | ~4.0s | ~1.5s |
| AST 解析 (100 行) | <10ms | <10ms |

> ⚠️ 以上为实测参考值，实际受硬件影响。
